# 毕业设计项目讲解稿（功能实现框架与通信）

## 1. 项目总体实现框架
本项目采用分层解耦的实现框架，由四个核心子系统组成：
- 前端交互层（Web）
- 业务编排层（Server）
- 算法推理层（Model）
- 知识检索问答层（RAG）

设计思路是把“界面交互、业务流程、模型计算、知识服务”拆分成独立能力，通过标准接口通信，保证系统可维护、可扩展。

---

## 2. 各框架职责与技术选型

### 2.1 前端交互层（Web）
职责：
- 提供检测、评估、报告、知识问答的统一操作界面
- 负责用户输入、结果展示、状态反馈

技术：
- Vue 3：组件化页面开发
- Vite：前端工程构建与开发服务器
- Element Plus：界面组件库
- Axios + Fetch：HTTP 请求与流式消息接收

### 2.2 业务编排层（Server）
职责：
- 接收前端请求，管理业务流程
- 维护任务状态、文件信息、评估结果
- 作为中台调度算法服务

技术：
- Spring Boot：REST API 与业务服务框架
- MyBatis-Plus：数据库访问与对象映射
- MySQL：任务、文件、结果等结构化数据存储

### 2.3 算法推理层（Model）
职责：
- 接收待检测数据并执行深度学习推理
- 输出检测掩膜、光谱特征等结果
- 执行评估指标计算
- 核心算法采用本人训练的 DMSSN/DSST 体系模型（论文框架）

模型框架（依据论文）：
- 第一阶段：Distilled Spectral Encoding（蒸馏光谱编码）
- 使用 GMM 光谱均质化作为预处理，降低背景复杂度并增强目标与背景光谱对比
- 构建 Teacher-Student 自编码器，先训练深层 Teacher，再通过知识蒸馏训练轻量 Student
- 在保证编码质量的同时降低推理开销，实现高光谱降维与关键信息保留
- 第二阶段：MSST（Mixed Spectral-Spatial Transformer）特征提取
- 由多层 MSS Block 堆叠组成
- 每个 MSS Block 采用分组注意力头：一组聚焦光谱特征，一组聚焦空间特征
- 通过谱-空联合建模与多尺度特征融合，输出更精细的显著目标响应
- 第三阶段：Saliency Prediction（显著图预测）
- 将 MSST 输出特征映射为最终显著性结果，用于后续评估与报告生成

技术：
- Flask：轻量推理服务接口
- PyTorch：自训练 DMSSN/DSST 模型的训练、蒸馏与推理
- NumPy/科学计算工具：数据处理与指标计算

### 2.4 知识检索问答层（RAG）
职责：
- 对论文与文档构建向量索引
- 根据问题检索相关内容并生成回答
- 支持对话记忆和流式输出

技术：
- Flask：RAG API 服务
- LangChain：检索与生成链路编排
- Chroma：向量数据库
- Ollama：本地 LLM 与 Embedding 模型服务
- SSE：流式输出问答内容

## 3. 框架之间的通信机制

### 3.1 Web -> Server（主业务通信）
通信方式：HTTP/REST（JSON）
- 检测请求：前端提交任务参数
- 评估请求：前端提交真值与任务信息
- 报告请求：前端请求导出文件

作用：
- 由后端统一承接业务，前端只负责交互
- 降低前端与算法服务的直接耦合

### 3.2 Server -> Model（算法调用通信）
通信方式：HTTP/REST（服务间调用）
- 后端将文件绝对路径等参数传给模型服务
- 模型服务调用自训练 DSST 模型完成推理并返回检测结果与评估指标

作用：
- 形成“业务层调度算法层”的结构
- 模型可独立迭代，不影响前端框架

### 3.3 Web -> RAG（知识问答通信）
通信方式：
- 普通接口：HTTP/REST
- 问答接口：SSE 流式通信

作用：
- 问答可边生成边展示，提高交互体验
- 支持展示来源文档和回答过程

### 3.4 Server <-> Database（数据持久化通信）
通信方式：ORM/Mapper 持久化
- 保存任务状态、文件元数据、评估结果
- 支撑结果追踪、报告生成、历史查询

## 4. 功能实现主流程（端到端）

### 流程 A：目标检测
1. 前端发起检测请求到业务后端
2. 后端创建或更新任务状态
3. 后端调用模型服务执行推理
4. 模型返回结果数据
5. 后端入库并返回前端展示

### 流程 B：效果评估
1. 前端提交评估请求
2. 后端整理预测结果与真值数据
3. 调用模型服务计算指标
4. 后端持久化指标并返回

### 流程 C：RAG 问答
1. 前端提交问题到 RAG 服务
2. RAG 先检索向量库相关内容
3. LLM 基于检索上下文生成回答
4. 通过 SSE 逐段返回给前端实时展示

---

## 5. 为什么采用这种框架
- 分层清晰：每层边界明确，职责单一
- 易于维护：某一层升级不会大范围影响其他层
- 易于扩展：可替换模型、增加知识库、扩展新页面
- 易于演示：完整体现“输入 -> 处理 -> 输出 -> 解释”的系统能力

---

## 6. 项目总结（技术实现角度）
本项目的核心不是单一算法，而是用工程化框架把多个技术栈协同起来：
- 用前端框架实现可交互界面
- 用后端框架完成业务编排与数据管理
- 用 Python 推理服务承载算法能力
- 用 RAG 框架增强知识解释能力

最终形成了一个具备业务闭环、技术解耦和可扩展性的高光谱智能系统。
